# Adversarial Code Review: Defending Against AI-Generated Code Echo Chambers

**Date:** 2026-03-28

## Executive Summary

Adversarial code review is a verification pattern that deploys a separate, independent AI agent to review and critique code generated by another agent or developer, breaking the "echo chamber" where a model self-validates its own output. This becomes critical as autonomous code generation scales: models are measurably bad at self-review (hallucinating correctness, missing edge cases), and standard code review processes weren't designed to catch AI-specific failure modes.

The key insight: **a model that generated code will hallucinate correctness when asked to review its own output.** The adversarial pattern solves this by using a separate session, model, or specialized agent architected explicitly to be skeptical, shifting from "does this code work?" to "can I break this code?"

This guide covers the problem, the landscape of tools and patterns, implementation strategies for Claude Code and CI/CD, autonomous remediation, metrics, anti-patterns, and a decision framework for when to deploy adversarial review.

## Part 1: The Problem

### Why Self-Review Fails

**Echo Chamber Effect:** When a language model generates code and is then asked to review it, the model exhibits strong validation bias—it tends to confirm that the code is correct, even when it objectively isn't. This is not a safety filter; it's a fundamental property of how LLMs work. The model's training has reinforced patterns of "generate plausible output → validate that output," creating a coherence bias that prevents self-correction on logical errors.

Sources:
- [MIT Sloan: "When AI Gets It Wrong: Addressing AI Hallucinations and Bias"](https://mitsloanedtech.mit.edu/ai/basics/addressing-ai-hallucinations-and-bias/)
- [DiffRay: "LLM Hallucinations in AI Code Review"](https://diffray.ai/blog/llm-hallucinations-code-review/)
- [Nature (2025): "My AI is Lying to Me: User-reported LLM hallucinations in AI"](https://www.nature.com/articles/s41598-025-15416-8)

**Hallucination Rates in AI-Generated Code:**
- 29–45% of AI-generated code contains security vulnerabilities
- Nearly 20% of package recommendations point to libraries that don't exist
- Models show strong bias toward labeling packages as "valid" even when they're fabricated
- Chain-of-Verification (CoVe) techniques can reduce hallucination by up to 96%, but require a separate verification agent

Sources:
- [USENIX Security 2025: "We Have a Package for You! A Comprehensive Analysis of Package Hallucinations"](https://www.usenix.org/system/files/conference/usenixsecurity25/sec25cycle1-prepub-742-spracklen.pdf)
- [PromptQuorum: "AI Code Review: Tools, Hallucination Rates, and Verification"](https://www.promptquorum.com/prompt-engineering/ai-code-review)
- [The Cyber Instructor: "The Verified Output Rule: Defending Against AI Hallucinations in Code"](https://thecyberinstructor.substack.com/p/the-verified-output-rule-defending)

**AI-Generated Code Blind Spots:**
- Missing edge cases and error paths (null checks, race conditions)
- Unverified external API calls (APIs that don't exist, wrong endpoints)
- Logic gaps that compile but don't execute correctly
- Security issues: SQL injection, XSS, broken authentication, insecure deserialization
- No production context awareness (doesn't understand service dependencies, monitoring)

### What Standards Review Misses

Traditional code review processes assume human reviewers: they focus on architecture, style, maintainability, and "Is this the right solution?" They don't focus on catching the specific failure modes of LLM generation:
- Hallucinated library names and APIs
- Subtle logical inconsistencies that "look right"
- Edge cases the training data didn't emphasize
- Security patterns the model saw less frequently in training

Result: Human reviewers catch ~60% of defects when doing formal inspection, but for AI-generated code, that rate is lower because reviewers aren't primed to expect hallucinations and logical gaps.

## Part 2: What Adversarial Review Is

### The Critic Agent Pattern

**Adversarial code review deploys a separate AI session—the Critic Agent—to review code produced by the Builder Agent.** The Critic is architected explicitly to be skeptical, adversarial by design.

Core components:

1. **Builder Agent:** Optimized for speed, syntax correctness, and generating working code quickly. Biased toward "this should work."

2. **Critic Agent:** Optimized for reasoning, logic validation, and vulnerability detection. Biased toward "how can I break this?" Has explicit instructions to be skeptical and flag even minor inconsistencies.

3. **Spec/Contract:** The shared source of truth. Both agents use the Spec to define requirements, constraints, and quality criteria. The Critic compares code strictly against the Spec.

4. **Review Gate:** Sits between automated tooling and human review. Outputs flagged by the Critic are prioritized for human review; issues that pass Critic validation are low-risk.

Source: [ASDLC.io: "Adversarial Code Review"](https://asdlc.io/patterns/adversarial-code-review/)

### The Spec as Shared Truth

The adversarial pattern only works if both Builder and Critic reference the same Spec. The Spec should contain:

- **Blueprint:** Explicit constraints (must use PostgreSQL, must support concurrent writes, must return in <100ms)
- **Contract:** Quality criteria (must handle errors, must log failures, must validate user input)
- **Edge Cases:** Specific scenarios the code must handle (null inputs, empty arrays, network timeouts, concurrent access)

When the Critic can't determine if code meets the Spec, that surfaces specification gaps—invaluable for hardening requirements before human review.

Source: [ASDLC.io: "AGENTS.md Specification: A Research-Backed Guide"](https://asdlc.io/practices/agents-md-spec/)

### Critic Constitution

The Critic Agent needs explicit framing to be adversarial. A strong Constitution includes:

> "You are skeptical. Your job is to reject code that violates the Spec, even if it 'works.' Favor false positives over false negatives. Assume edge cases exist unless explicitly handled. If a package name is used, verify it's real. If an API is called, verify the endpoint is correct. Don't assume—check."

This explicit adversarial framing overcomes the LLM's default bias toward validation and coherence.

Source: [ASDLC.io: "Adversarial Code Review"](https://asdlc.io/patterns/adversarial-code-review/)

## Part 4: Implementation Patterns

### Pattern 1: Manual Adversarial Review (Separate Claude Session)

**When to use:** For critical code, architectural decisions, security-sensitive changes. Low automation cost, high signal.

**Process:**

1. Builder completes code in one Claude Code session or IDE
2. Open a second Claude Code session with the Critic Constitution
3. Paste the code, Spec, and blueprint into the Critic session
4. Critic reviews against the Spec, flags violations, edge cases, security issues
5. Iterate: Builder fixes issues flagged by Critic
6. Once Critic approves, submit to human review (focused now on architecture/strategy)

**Configuration:** No config needed—just discipline to open a separate session and use the right prompt.

**Example Critic Prompt:**

```
You are a skeptical code reviewer. Your job is to identify violations of this Spec,
edge cases not handled, and potential security issues.

SPEC:
{spec_content_here}

CODE TO REVIEW:
{code_content_here}

REVIEW CRITERIA:
1. Does the code implement every requirement in the Spec?
2. Are all constraints respected (database choice, concurrency model, etc.)?
3. Are all edge cases handled (null inputs, empty arrays, network failures)?
4. Are external APIs verified to exist?
5. Are there security vulnerabilities (injection, auth, deserialization)?
6. Is error handling comprehensive?

Be skeptical. Favor false positives over false negatives.
Flag even minor inconsistencies.
```

### Pattern 2: Automated Adversarial Review in Claude Code (PreToolUse/PostToolUse Hooks)

**When to use:** Teams using Claude Code daily. Enforces adversarial review on every file change.

**How it works:**

1. Configure a PreToolUse hook that intercepts file writes
2. Before Claude writes code, the hook sends it to a background Critic session
3. If Critic flags issues, the hook surfaces them (can warn or block)
4. PostToolUse runs tests/linting after writes

**Configuration:** `.claude/settings.json` at project root

```json
{
  "hooks": {
    "PreToolUse": {
      "write_file": {
        "action": "warn",
        "command": "critic-review",
        "options": {
          "model": "claude-opus",
          "constitution": "skeptical",
          "spec_path": ".claude/SPEC.md",
          "severity_threshold": "medium"
        }
      }
    },
    "PostToolUse": {
      "write_file": {
        "action": "notify",
        "command": "lint-and-test"
      }
    }
  }
}
```

**Implementation Notes:**
- Hook commands are custom—you define them in your Claude Code plugin or skill
- "warn" allows override; "block" prevents the action
- Severity threshold filters noise (don't warn on style issues, only logic/security)
- Runs asynchronously to avoid blocking the Builder

Sources:
- [Claude Code Docs: "Automate workflows with hooks"](https://code.claude.com/docs/en/hooks-guide)
- [PixelMojo: "Claude Code Hooks: PreToolUse, PostToolUse & All 12"](https://www.pixelmojo.io/blogs/claude-code-hooks-production-quality-ci-cd-patterns)

### Pattern 3: CI/CD Integration (GitHub Actions / GitLab CI)

**When to use:** Every pull request should get adversarial review before human review.

**Process:**

1. Trigger on `pull_request` event
2. Fetch the PR diff and full repo context
3. Call Claude Code Review API or custom Critic script
4. Post findings as GitHub checks or comments
5. Block merge if findings exceed severity threshold (configurable)

**GitHub Actions Example:**

```yaml
name: Adversarial Code Review

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  adversarial-review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
      checks: write

    steps:
      - name: Checkout code
        uses: actions/checkout@v3
        with:
          fetch-depth: 0

      - name: Get PR diff
        id: diff
        run: |
          git fetch origin ${{ github.base_ref }}
          git diff origin/${{ github.base_ref }}...HEAD > /tmp/changes.diff

      - name: Run Adversarial Critic Review
        id: review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          PR_DIFF: ${{ steps.diff.outputs.diff }}
          SPEC_PATH: ./SPEC.md
        run: |
          python3 scripts/critic_review.py \
            --diff /tmp/changes.diff \
            --spec ./SPEC.md \
            --severity threshold=medium \
            --output json > /tmp/findings.json

      - name: Post findings to PR
        uses: actions/github-script@v6
        with:
          script: |
            const fs = require('fs');
            const findings = JSON.parse(fs.readFileSync('/tmp/findings.json', 'utf8'));

            // Post as review comment if high severity
            if (findings.high_severity.length > 0) {
              github.rest.pulls.createReview({
                owner: context.repo.owner,
                repo: context.repo.repo,
                pull_number: context.issue.number,
                event: 'REQUEST_CHANGES',
                body: `## Adversarial Code Review Findings\n\n${findings.summary}`,
                comments: findings.comments
              });
            }

      - name: Enforce review gate
        if: failure()
        run: |
          echo "Code review failed adversarial checks. Fix and push again."
          exit 1
```

**Critic Script (Python example):**

```python
import anthropic
import json
import sys
from pathlib import Path

def run_critic_review(diff: str, spec: str, severity_threshold: str) -> dict:
    client = anthropic.Anthropic()

    prompt = f"""You are a skeptical code reviewer. Review this PR diff against the Spec.

SPEC:
{spec}

PR DIFF:
{diff}

OUTPUT FORMAT (JSON):
{{
  "summary": "Brief summary of findings",
  "high_severity": [
    {{"line": 42, "issue": "...", "suggestion": "..."}},
  ],
  "medium_severity": [...],
  "low_severity": [...]
}}

Be specific. Reference line numbers. Favor false positives over false negatives.
"""

    message = client.messages.create(
        model="claude-opus-4-1",
        max_tokens=2000,
        messages=[
            {"role": "user", "content": prompt}
        ]
    )

    # Extract JSON from response
    response_text = message.content[0].text
    # Parse JSON (add error handling as needed)
    findings = json.loads(response_text)

    return findings

if __name__ == "__main__":
    diff_path = sys.argv[sys.argv.index("--diff") + 1]
    spec_path = sys.argv[sys.argv.index("--spec") + 1]

    with open(diff_path) as f:
        diff = f.read()
    with open(spec_path) as f:
        spec = f.read()

    findings = run_critic_review(diff, spec, "medium")
    print(json.dumps(findings, indent=2))
```

### Pattern 4: Multi-Model Adversarial Review

**When to use:** Critical systems (payments, auth, data migrations) where comprehensive coverage is worth the cost.

**Process:**

1. Submit code to multiple models in parallel (Claude, GPT-4o, Gemini)
2. Each model reviews independently
3. Synthesize findings: High confidence = issues multiple models flagged
4. Low confidence = single-model findings (surface but don't block)

**Configuration:**

```yaml
# .claude/adversarial-review.yml
critic_models:
  - provider: anthropic
    model: claude-opus-4-1
    constitution: skeptical
  - provider: openai
    model: gpt-4o
    constitution: skeptical
  - provider: google
    model: gemini-2-0
    constitution: skeptical

consensus_rules:
  high_confidence: 2+ models agree
  medium_confidence: 1 model flags
  ignore_single_model_style_issues: true

severity_mapping:
  security: high
  logic_error: high
  missing_edge_case: medium
  style: low
```

**Multi-Model Debate Pattern:** Have models critique each other:

```
Model A generates findings.
Model B reviews Model A's findings for false positives.
Model C synthesizes consensus findings.
Only consensus findings go to human review.
```

### Pattern 5: Specialist Swarm Review

**When to use:** Large teams, complex systems with many failure modes (security, performance, architecture, scale).

**Architecture:** Deploy 4–7 specialist agents in parallel, each hunting a different class of issue.

**Specialist Types:**

| Specialist | Looks For |
|-----------|-----------|
| **Security Auditor** | SQL injection, XSS, auth bypass, deserialization, crypto mistakes, secrets in logs |
| **Performance Analyst** | N+1 queries, memory leaks, algorithmic complexity, unbounded loops, blocking I/O |
| **Architect** | Violates design patterns, breaks service contracts, missing error handling, architectural coherence |
| **API Reviewer** | External APIs called without verification, wrong endpoints, missing error codes, rate limits |
| **Concurrency Expert** | Race conditions, deadlocks, ordering issues, incomplete synchronization |
| **Logic Validator** | State machine violations, invariant violations, off-by-one errors, missing edge cases |
| **Synthesizer** | Combines findings, removes duplicates, ranks by severity, surfaces to human |

**Configuration:**

```yaml
swarm:
  topology: parallel
  agents:
    - id: security_auditor
      role: Security Auditor
      prompt: |
        You hunt security vulnerabilities: SQL injection, XSS, auth bypass,
        broken crypto, exposed secrets. Assume all input is hostile.

    - id: performance_analyst
      role: Performance Analyst
      prompt: |
        You optimize for scale. Identify N+1 queries, memory leaks,
        O(n²) algorithms, blocking I/O, unbounded growth.

    - id: architect
      role: Architect
      prompt: |
        You enforce design patterns and contracts. Flag violations of
        service interfaces, missing error handling, architectural misalignment.

    - id: synthesizer
      role: Synthesizer
      prompt: |
        You combine findings from all specialists. Remove duplicates.
        Rank by severity. Surface high-confidence findings to human review.

  orchestration:
    execution: parallel
    timeout_per_agent: 30s
    vote_threshold: 2  # 2+ agents must agree for high confidence
```

**Execution Flow:**

```
Code → [Security Auditor, Performance Analyst, Architect, Concurrency Expert] (parallel)
         ↓
      [Synthesizer] (waits for all, combines findings)
         ↓
      [Human Reviewer] (high-confidence findings only)
```

## Part 5: Autonomous Remediation

### What Autonomous Remediation Is

Instead of just flagging issues, autonomous remediation agents:
1. Detect issues (same as review)
2. Generate candidate fixes
3. Validate fixes (testing, SAST, integration tests)
4. Open a pull request or patch automatically
5. Explain every change (audit trail)

This is the next frontier: moving from "fix it yourself" to "the system fixed it for you."

### How It Works

**Example:** Snyk Agent Fix, Harness AI, Orca

1. **Detection:** Vulnerability or logic error found in code (via SAST, dependency scanning, or manual report)
2. **Generation:** AI generates a fix (e.g., sanitize input for XSS, add rate limit, add null check)
3. **Validation:** Run the fix through:
   - Static analysis (SAST, linting)
   - Dependency checks (SCA)
   - Integration tests
   - Fuzzing
   - Policy checks
4. **PR Creation:** If validation passes, open a PR with:
   - Detailed explanation of the vulnerability
   - Explanation of the fix
   - Test results
   - Link to original issue
5. **Human Approval:** Developer reviews and merges (or rejects)

### Implementation Pattern

```yaml
# .github/workflows/auto-remediation.yml
name: Autonomous Vulnerability Remediation

on:
  pull_request:
    types: [opened, synchronize]
  workflow_dispatch:

jobs:
  detect_and_remediate:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      security-events: read

    steps:
      - name: Checkout
        uses: actions/checkout@v3

      - name: Run SAST
        id: sast
        run: |
          # Run your SAST tool (Semgrep, SonarQube, etc.)
          semgrep --config=p/owasp-top-ten --json > /tmp/sast-findings.json

      - name: Run Dependency Scan
        id: deps
        run: |
          # Run your SCA tool (Snyk, Dependabot, etc.)
          snyk test --json > /tmp/deps-findings.json

      - name: Merge findings
        id: merge
        run: |
          python3 - << 'EOF'
          import json
          with open('/tmp/sast-findings.json') as f:
              sast = json.load(f)
          with open('/tmp/deps-findings.json') as f:
              deps = json.load(f)

          all_findings = sast.get('results', []) + deps.get('vulnerabilities', [])

          # Filter to auto-remediable issues
          remediable = [f for f in all_findings if f.get('remediable') == True]

          with open('/tmp/remediable.json', 'w') as f:
              json.dump(remediable, f)

          print(f"Found {len(remediable)} remediable issues")
          EOF

      - name: Generate fixes
        id: fixes
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          python3 - << 'EOF'
          import json
          import anthropic

          with open('/tmp/remediable.json') as f:
              findings = json.load(f)

          client = anthropic.Anthropic()
          fixes = []

          for finding in findings:
              prompt = f"""Generate a minimal, production-safe fix for this vulnerability:

              VULNERABILITY: {finding['type']}
              LOCATION: {finding['file']}:{finding['line']}
              DESCRIPTION: {finding['message']}

              Provide a code patch. Be minimal. No refactoring unless required for the fix.
              Output ONLY the corrected code snippet, no explanation.
              """

              msg = client.messages.create(
                  model="claude-opus-4-1",
                  max_tokens=1000,
                  messages=[{"role": "user", "content": prompt}]
              )

              fixes.append({
                  'finding_id': finding['id'],
                  'file': finding['file'],
                  'line': finding['line'],
                  'fix': msg.content[0].text
              })

          with open('/tmp/fixes.json', 'w') as f:
              json.dump(fixes, f)
          EOF

      - name: Validate fixes
        id: validate
        run: |
          # Run tests, SAST, linting on the fixed code
          # For each fix:
          #   1. Apply it to a temp branch
          #   2. Run test suite
          #   3. Run SAST again (should be clean)
          #   4. Mark as validated or rejected

          python3 - << 'EOF'
          import json
          import subprocess

          with open('/tmp/fixes.json') as f:
              fixes = json.load(f)

          validated = []

          for fix in fixes:
              # Create temp branch, apply fix, test
              result = subprocess.run([
                  'git', 'checkout', '-b', f"fix/{fix['finding_id']}"
              ], capture_output=True)

              # Apply the fix (implementation depends on file type)
              # Run tests
              test_result = subprocess.run(['npm', 'test'], capture_output=True)

              if test_result.returncode == 0:
                  validated.append(fix)

              # Cleanup
              subprocess.run(['git', 'checkout', '-'], capture_output=True)

          with open('/tmp/validated-fixes.json', 'w') as f:
              json.dump(validated, f)
          EOF

      - name: Create PR for each validated fix
        uses: actions/github-script@v6
        with:
          script: |
            const fs = require('fs');
            const validated = JSON.parse(
              fs.readFileSync('/tmp/validated-fixes.json', 'utf8')
            );

            for (const fix of validated) {
              // Create a branch for this fix
              await exec.exec('git', ['checkout', '-b', `fix/${fix.finding_id}`]);

              // Apply the fix (simplified—real implementation would parse and rewrite files)
              // ...

              // Commit and push
              await exec.exec('git', ['add', '.']);
              await exec.exec('git', ['commit', '-m',
                `Fix vulnerability: ${fix.finding_id}`]);
              await exec.exec('git', ['push', 'origin', `fix/${fix.finding_id}`]);

              // Open PR
              await github.rest.pulls.create({
                owner: context.repo.owner,
                repo: context.repo.repo,
                head: `fix/${fix.finding_id}`,
                base: context.ref,
                title: `[Auto-Fix] Vulnerability in ${fix.file}:${fix.line}`,
                body: `Automated fix for vulnerability detected in code review.

                **Validated:** Tests passed, SAST clean, ready for review.`,
                labels: ['auto-remediation', 'security']
              });
            }
```

### Key Constraints

**Only auto-remediate issues that are:**
1. Self-contained (don't require changes across multiple files)
2. Have canonical fixes (input validation, rate limiting, null checks)
3. Pass all tests after the fix
4. Don't break the API contract

**Never auto-remediate:**
- Architectural changes
- Business logic rewrites
- Database schema changes
- Permission/access control changes (humans must review)

## Part 6: What Reviewers Should Focus On

Adversarial reviewers should be skeptical about categories that self-review misses:

### 1. **Hallucinated APIs and Packages**

Check every external call:
- Does the package exist on npm/PyPI/crates.io/Maven Central?
- Does the function/method exist on that package version?
- Are the arguments in the correct order?
- Is the endpoint URL correct?

**Review Prompt:**
```
For every external API call or package import:
1. Verify the package exists in the appropriate registry
2. Verify the function/method exists
3. Verify the arguments match the signature
4. Flag as ERROR if any cannot be verified
```

### 2. **Unhandled Edge Cases**

Assume the code will receive adversarial or boundary input:
- Null / undefined / empty collections
- Maximum/minimum numeric values
- Concurrent access to shared state
- Network timeouts and retries
- Rate limits and backpressure

**Review Prompt:**
```
For every function, ask:
1. What happens if input is null?
2. What happens if input is empty?
3. What happens if input is maximum size?
4. What if the operation times out?
5. What if called concurrently?

Flag any case not explicitly handled.
```

### 3. **Security Vulnerabilities**

Standard OWASP categories, but applied to AI-generated code:
- SQL injection (parameterized queries)
- XSS (escaping/sanitization)
- Auth bypass (permission checks before sensitive ops)
- Exposed secrets (no credentials in code)
- Unsafe deserialization
- Insecure crypto (never roll your own)

**Review Prompt:**
```
Security Check:
1. Are all database queries parameterized (no string interpolation)?
2. Is user input escaped before output (HTML, SQL, shell)?
3. Are permission checks present before sensitive operations?
4. Are credentials ever hardcoded or logged?
5. Is serialized data validated before deserialization?
6. Is cryptography delegated to libraries (not custom)?
```

### 4. **Logic Errors and Invariant Violations**

These are subtle but critical:
- Off-by-one errors in loops
- State machine violations (wrong order of operations)
- Race conditions (shared state modifications)
- Incorrect conditionals (e.g., `>` instead of `>=`)

**Review Prompt:**
```
Logic Check:
1. Do loop bounds match the intent (inclusive vs exclusive)?
2. Is the state machine order correct?
3. Could concurrent access violate invariants?
4. Are all conditionals correct (>, >=, <, <=)?
5. Are all required conditions checked before proceeding?
```

### 5. **Missing Error Handling**

Every fallible operation needs handling:
- Network calls (timeout, 4xx, 5xx)
- File I/O (not found, permission denied)
- Parsing (malformed input)
- Resource limits (quota exceeded)

**Review Prompt:**
```
Error Handling Check:
For every fallible operation (network, file, parse, DB):
1. Is there a try/catch or error return?
2. Is the error case handled meaningfully (not just logged)?
3. Is the error propagated or recovered from appropriately?
4. Would a human understand what went wrong?
```

### 6. **Performance and Scalability**

- N+1 query patterns (loading one item triggers N queries)
- Unbounded loops or recursion
- Memory leaks (caches not evicted)
- Blocking I/O on hot paths

**Review Prompt:**
```
Performance Check:
1. Are there any N+1 query patterns?
2. Are loops/recursion bounded?
3. Are caches properly evicted?
4. Is I/O blocking? Could it be async?
5. Is the algorithm polynomial or exponential in input size?
```

## Part 7: Measuring Effectiveness

### Key Metrics

**1. Defect Detection Rate (DDR)**

```
DDR = (Defects found in review) / (Total defects that would reach production)

Target: 70%+ for adversarial review
Benchmark: 60% for formal human inspection, <50% for informal review
```

Measure by running adversarial review on a branch, then tracking which issues would have escaped to production if not caught.

**2. False Positive Rate (FPR)**

```
FPR = (Issues flagged that aren't real issues) / (Total issues flagged)

Target: <5% for adversarial review
Claude Code Review: <1%
```

False positives cause review fatigue. Track them and refine the Critic prompt.

**3. Time to First Feedback (TTFF)**

```
TTFF = Time from PR open to first review feedback

Target: <5 minutes for automated + <30 minutes for human
Benchmark: Claude Code Review ~20 minutes
```

**4. Review Throughput**

```
PRs reviewed per day / reviewer
% of PRs that pass first Critic review (no iteration)
```

**5. Defect Escape Rate**

```
Escape Rate = (Defects that reach production) / (Defects in PRs reviewed)

Track over time. Should decrease as Critic improves.
```

### Measurement Process

1. **Instrument CI/CD:** Log every finding, every false positive, every defect that escapes
2. **Correlate with production incidents:** When a production bug occurs, did adversarial review miss it? (If yes, retrain the Critic)
3. **A/B test:** Run adversarial review on 50% of PRs, measure defect escape for both groups
4. **Survey reviewers:** "Does this save you time?" "Do the findings feel accurate?"

Sources:
- [GeeksforGeeks: "Inspection Metrics in Software Engineering"](https://www.geeksforgeeks.org/inspection-metrics-in-software-engineering/)
- [ArXiv: "Towards Practical Defect-Focused Automated Code Review"](https://arxiv.org/html/2505.17928)

## Part 8: Anti-Patterns to Avoid

### Anti-Pattern 1: Over-Trusting AI Review

**Problem:** Treating "Critic passed" as "code is safe to merge."

**Reality:** AI reviewers are fast and catch common patterns, but they have blind spots (context, domain knowledge, business rules). They excel at finding bugs; they don't understand intent.

**Mitigation:**
- Keep human review in the gate (review is complementary, not replacement)
- Use adversarial review to triage high-confidence issues for human focus
- Mark low-confidence findings as "informational" (don't block)

### Anti-Pattern 2: Review Fatigue

**Problem:** Critic flags 50 issues per PR. Reviewers dismiss all of them.

**Reality:** Too much noise = reviewers ignore everything.

**Mitigation:**
- Tune the Critic Constitution to favor precision over recall
- Filter low-severity findings (nitpicks, style)
- Group findings by severity/category
- Provide metrics (e.g., "94% of flagged issues are real")

### Anti-Pattern 3: Too Many Reviewers

**Problem:** 7 specialist agents reviewing, each with different opinions. Developers paralyzed.

**Reality:** Consensus is hard. More reviewers = more conflict = slower shipping.

**Mitigation:**
- Limit swarm to 3–4 specialists for most code
- Use a Synthesizer agent to resolve conflicts
- Establish clear precedence (Security > Performance > Style)
- Require 2+ agents to agree for an issue to be flagged

### Anti-Pattern 4: Ignoring Disagreement

**Problem:** Critic disagrees with Builder. Ignored or overridden without justification.

**Reality:** Disagreement is signal. It means the code is unclear or ambiguous.

**Mitigation:**
- Treat disagreement as a spec issue (clarify the spec)
- Require explicit override + comment (audit trail)
- Track overrides—if a pattern repeats, update the Spec

### Anti-Pattern 5: Static Critic Constitution

**Problem:** Same Critic rules for all code (tests, infra, business logic).

**Reality:** Security-critical code needs different standards than utility functions.

**Mitigation:**
- Vary Critic Constitution by code type (security, infra, business logic, test)
- Use labels/tags in the repo to route to right Critic
- Adjust severity thresholds per domain

### Anti-Pattern 6: No Feedback Loop

**Problem:** Findings are logged, but not used to improve the Critic or fix systemic issues.

**Reality:** Critic gets stuck finding the same issues in every PR.

**Mitigation:**
- Monthly Critic audit: What did it miss? What was wrong?
- Retrain on recent false positives
- Update the Spec based on recurring ambiguities
- Share learnings across teams

## Part 9: Decision Framework: When to Deploy Adversarial Review

### Use Adversarial Review If

**YES** if:
- [ ] Code is AI-generated (Claude, Copilot, other LLM)
- [ ] Code touches security, payments, auth, or PII
- [ ] Code is performance-critical (scale matters)
- [ ] Code has complex business logic (state machines, workflows)
- [ ] Team has >5 engineers (ROI on automation is clear)
- [ ] Risk of production incidents is high

**MAYBE** if:
- [ ] Low-risk utilities (string formatting, simple math)
- [ ] High-churn code (feature flags, experiments)
- [ ] Team is <3 engineers (overhead may not pay off)
- [ ] Deadline is imminent (don't add friction now)

**NO** if:
- [ ] Code is auto-generated from templates (no review needed)
- [ ] Code is test-only (different standards)
- [ ] Code is documentation or config only
- [ ] Budget doesn't exist for the tooling

### Which Pattern to Start With

**For Solo Developer or Small Team (<5):**
Start with Pattern 1: **Manual Adversarial Review (Separate Session)**
- No setup, free, immediately actionable
- Opens second Claude session, paste code + spec, get feedback
- Cost: 5–10 minutes per PR

**For Growing Team (5–20):**
Graduate to Pattern 2: **Claude Code Hooks (PreToolUse/PostToolUse)**
- Automate the manual process
- Still Claude-based (no new tools)
- Cost: ~$50–200/month (depending on volume)

**For Enterprise (20+) or Risk-Critical Systems:**
Invest in Pattern 3: **CI/CD Integration + Multi-Model Review**
- Fully automated, always-on
- Multi-model consensus for high confidence
- Cost: ~$500–2000/month (depends on PR volume)

**For Mission-Critical Systems:**
Deploy Pattern 5: **Specialist Swarm Review**
- Multiple agents, each hunting specific issues
- Comprehensive coverage
- Cost: ~$1000–5000/month

### Sizing the Decision

```
Decision Tree:

Risk Level?
├─ Critical (payments, auth, PII) → Go to Pattern 3 or 5
├─ High (business logic, performance) → Go to Pattern 2 or 3
├─ Medium (features, backend) → Go to Pattern 1 or 2
└─ Low (docs, utilities) → Manual review only

Team Size?
├─ Solo → Pattern 1
├─ 3–10 → Pattern 1 or 2
├─ 10–50 → Pattern 2 or 3
└─ 50+ → Pattern 3 or 5

Budget?
├─ $0/month → Pattern 1
├─ <$100 → Pattern 2
├─ <$500 → Pattern 3
└─ Unlimited → Pattern 5
```

### GitHub (CI/CD Integration, Pattern 3)

**Step 1: Create Critic Script**

Save as `scripts/critic_review.py`:

```python
#!/usr/bin/env python3
import sys
import os
import json
import subprocess
import anthropic

def get_pr_diff():
    """Get the diff for the current PR."""
    result = subprocess.run(
        ["git", "diff", "origin/main...HEAD"],
        capture_output=True, text=True
    )
    return result.stdout

def get_spec():
    """Load the SPEC.md file."""
    if os.path.exists("SPEC.md"):
        with open("SPEC.md") as f:
            return f.read()
    return "No SPEC.md found"

def run_critic_review():
    """Run adversarial code review."""
    diff = get_pr_diff()
    spec = get_spec()

    client = anthropic.Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

    prompt = f"""You are a skeptical code reviewer for this project.

PROJECT SPEC:
{spec}

PR DIFF:
{diff}

OUTPUT FORMAT (valid JSON):
{{
  "summary": "Brief summary of findings",
  "findings": [
    {{
      "severity": "high|medium|low",
      "category": "security|logic|edge_case|performance|api|other",
      "line": "line_number_or_range",
      "issue": "What's wrong",
      "suggestion": "How to fix",
      "confidence": 0.0-1.0
    }}
  ],
  "overall_pass": true|false
}}

Review against the Spec. Be skeptical. Assume edge cases exist unless explicitly handled.
Favor false positives over false negatives. Output only valid JSON, no other text.
"""

    message = client.messages.create(
        model="claude-opus-4-1",
        max_tokens=2000,
        messages=[
            {"role": "user", "content": prompt}
        ]
    )

    response_text = message.content[0].text

    # Parse JSON from response
    try:
        findings = json.loads(response_text)
    except json.JSONDecodeError:
        # Try to extract JSON from response
        import re
        json_match = re.search(r'\{.*\}', response_text, re.DOTALL)
        if json_match:
            findings = json.loads(json_match.group())
        else:
            findings = {
                "summary": "Could not parse response",
                "findings": [],
                "overall_pass": False
            }

    return findings

if __name__ == "__main__":
    findings = run_critic_review()
    print(json.dumps(findings, indent=2))
```

**Step 2: Create GitHub Action**

Save as `.github/workflows/adversarial-review.yml`:

```yaml
name: Adversarial Code Review

on:
  pull_request:
    types: [opened, synchronize, reopened]

permissions:
  contents: read
  pull-requests: write
  checks: write

jobs:
  critic-review:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v3
        with:
          fetch-depth: 0

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: |
          pip install anthropic

      - name: Run Adversarial Critic Review
        id: review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          python3 scripts/critic_review.py > /tmp/findings.json
          cat /tmp/findings.json

      - name: Post findings to PR
        uses: actions/github-script@v6
        if: always()
        with:
          script: |
            const fs = require('fs');
            const findings = JSON.parse(fs.readFileSync('/tmp/findings.json', 'utf8'));

            if (findings.findings.length === 0) {
              github.rest.issues.createComment({
                issue_number: context.issue.number,
                owner: context.repo.owner,
                repo: context.repo.repo,
                body: '✅ Adversarial review passed. No issues found.'
              });
              return;
            }

            let summary = `## Adversarial Code Review Findings\n\n${findings.summary}\n\n`;

            // Group by severity
            const highSeverity = findings.findings.filter(f => f.severity === 'high');
            const mediumSeverity = findings.findings.filter(f => f.severity === 'medium');
            const lowSeverity = findings.findings.filter(f => f.severity === 'low');

            if (highSeverity.length > 0) {
              summary += `### 🔴 High Severity (${highSeverity.length})\n`;
              highSeverity.forEach(f => {
                summary += `- **${f.category}** (Line ${f.line}): ${f.issue}\n  Suggestion: ${f.suggestion}\n`;
              });
              summary += '\n';
            }

            if (mediumSeverity.length > 0) {
              summary += `### 🟡 Medium Severity (${mediumSeverity.length})\n`;
              mediumSeverity.forEach(f => {
                summary += `- **${f.category}** (Line ${f.line}): ${f.issue}\n`;
              });
              summary += '\n';
            }

            if (lowSeverity.length > 0) {
              summary += `### 🟢 Low Severity (${lowSeverity.length})\n`;
              lowSeverity.forEach(f => {
                summary += `- **${f.category}** (Line ${f.line}): ${f.issue}\n`;
              });
              summary += '\n';
            }

            summary += '\n**Action:** Address high-severity issues before merge. Medium/low are informational.';

            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: summary
            });

            // Fail the check if high-severity findings
            if (highSeverity.length > 0) {
              core.setFailed(`Adversarial review found ${highSeverity.length} high-severity issues`);
            }
```

## Part 11: Integrating Multiple Models (Pattern 4)

### Multi-Model Setup

Create `scripts/multi_model_review.py`:

```python
#!/usr/bin/env python3
import json
import os
import asyncio
import anthropic
import openai
import google.generativeai as genai

async def review_with_claude(diff: str, spec: str) -> dict:
    """Review with Claude."""
    client = anthropic.Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

    message = client.messages.create(
        model="claude-opus-4-1",
        max_tokens=2000,
        messages=[{
            "role": "user",
            "content": f"""Review this code against the spec.

SPEC:
{spec}

CODE:
{diff}

Output JSON with: summary, findings (list), severity, category, suggestion."""
        }]
    )

    return json.loads(message.content[0].text)

async def review_with_gpt(diff: str, spec: str) -> dict:
    """Review with GPT-4o."""
    client = openai.OpenAI(api_key=os.getenv("OPENAI_API_KEY"))

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{
            "role": "user",
            "content": f"""Review this code against the spec.

SPEC:
{spec}

CODE:
{diff}

Output JSON with: summary, findings (list), severity, category, suggestion."""
        }],
        temperature=0.2
    )

    return json.loads(response.choices[0].message.content)

async def review_with_gemini(diff: str, spec: str) -> dict:
    """Review with Gemini."""
    genai.configure(api_key=os.getenv("GOOGLE_API_KEY"))
    model = genai.GenerativeModel('gemini-2.0-flash')

    response = model.generate_content(f"""Review this code against the spec.

SPEC:
{spec}

CODE:
{diff}

Output JSON with: summary, findings (list), severity, category, suggestion.""")

    return json.loads(response.text)

async def run_multi_model_review(diff: str, spec: str):
    """Run reviews in parallel, synthesize results."""

    results = await asyncio.gather(
        review_with_claude(diff, spec),
        review_with_gpt(diff, spec),
        review_with_gemini(diff, spec),
        return_exceptions=True
    )

    claude_findings = results[0].get('findings', []) if not isinstance(results[0], Exception) else []
    gpt_findings = results[1].get('findings', []) if not isinstance(results[1], Exception) else []
    gemini_findings = results[2].get('findings', []) if not isinstance(results[2], Exception) else []

    # Consensus: issues flagged by 2+ models
    all_findings = claude_findings + gpt_findings + gemini_findings

    # Group by category and location
    consensus = {}
    for finding in all_findings:
        key = (finding.get('category'), finding.get('line'))
        if key not in consensus:
            consensus[key] = {'count': 0, 'finding': finding, 'models': []}
        consensus[key]['count'] += 1
        consensus[key]['models'].append(finding.get('model', 'unknown'))

    # High confidence: 2+ models agree
    high_confidence = [c['finding'] for c in consensus.values() if c['count'] >= 2]
    low_confidence = [c['finding'] for c in consensus.values() if c['count'] == 1]

    return {
        'high_confidence': high_confidence,
        'low_confidence': low_confidence,
        'summary': f"{len(high_confidence)} consensus findings, {len(low_confidence)} single-model findings"
    }

if __name__ == "__main__":
    import subprocess

    diff = subprocess.run(["git", "diff", "origin/main...HEAD"],
                         capture_output=True, text=True).stdout

    with open("SPEC.md") as f:
        spec = f.read()

    results = asyncio.run(run_multi_model_review(diff, spec))
    print(json.dumps(results, indent=2))
```
