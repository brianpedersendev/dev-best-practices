## 2. Contradictions Between Files

### 2a. When to use agents
- `ai-native-architecture.md` warns: *"Using agents when a simple function call suffices. Not everything needs autonomous reasoning."*
- `swarm-patterns-by-dev-stage.md` recommends multi-agent patterns for **every** dev stage (research, planning, coding, testing, review, debugging, docs, deployment, maintenance)
- **No reconciliation** — a reader following both gets conflicting advice

### 2b. Optimization sequencing
- `ai-native-architecture.md`: Start with prompt engineering → add RAG only if accuracy plateaus
- `cost-optimization-playbook.md`: Start with model routing → then caching
- `prompt-engineering-patterns.md`: Structured prompts as the default, RAG as enhancement
- **No unified decision tree** across documents

### 2c. Cache effectiveness
- `ai-native-architecture.md`: Prompt caching gives "90% discount"
- `cost-optimization-playbook.md`: Semantic caching "40-70% hit rate", prompt caching "20-50% savings"
- **Unclear** whether these are the same mechanism or complementary strategies

### 2d. Evaluation thresholds
- `ai-native-architecture.md`: "Gate deployment if accuracy drops >3%"
- `testing-ai-generated-code.md`: "Measure accuracy on 100+ diverse examples" but no threshold
- **Unresolved:** Is 3% on 100 samples (3 failures) statistically significant?

**Recommendation:** Create a unified "Decision Trees" section or doc that resolves sequencing and threshold conflicts.

## 3. Overconfident Claims (Should Be Hedged)

| Claim | Issue |
|-------|-------|
| "50-70% spending reduction" (cost playbook title) | Best case. What's the median? Likely 20-40% for most teams |
| "TDD reduces AI code defects 40-90%" | Huge range with no guidance on what determines which end |
| "PostgreSQL first; specialist databases only at ceilings" | Assumes DBA expertise. Managed options (Pinecone, Qdrant) may be better for teams without it |
| "LLM-as-judge: 'Is this answer correct?'" | Presented as straightforward. Fails for subjective outputs, judge hallucination, style decisions |
| LangGraph recommended across multiple files | No guidance on when CrewAI or a simple loop is the better choice |
| "End-to-end latency: 2-5 min. Worth it because review time drops 70%" | The 70% is unsourced and likely varies dramatically by codebase |

**Recommendation:** Add ranges, conditions, and "depends on" qualifiers. State median outcomes, not just best-case.

## 4. Missing Major Topics

### 4a. Error Recovery & Fallback Patterns (HIGH priority)
No document covers: model failures, token limit hits, rate limits, timeouts, cascading fallbacks (Opus → Sonnet → Haiku), circuit breaker patterns for agentic systems. This is production-critical.

### 4b. When NOT to Use AI (HIGH priority)
Every document is pro-AI. No explicit guidance on tasks where traditional code is better, where AI adds risk without value, or when to abandon an AI approach.

### 4c. Multi-Model Governance (MEDIUM priority)
Teams using Claude Code + Cursor + Gemini in the same codebase have no playbook for consistency, configuration versioning, or handling tool divergence.

### 4d. CI/CD Integration Safety (MEDIUM priority)
How to safely run AI in existing pipelines without breaking tests, introducing flaky tests, or regressing performance. Teams are scared to automate this — no de-risking guide exists.

### 4e. Evaluation Beyond LLM-as-Judge (MEDIUM priority)
Only evaluation method documented. Missing: statistical significance for small samples, domain-expert review patterns, cost of evaluation vs. value, handling non-deterministic outputs.

### 4f. RAG Knowledge Staleness Detection (LOW priority)
RAG is covered extensively but no guidance on detecting when knowledge is stale, versioning knowledge, or detecting embedding drift.

### 4g. The "Death Valley" of Integration (LOW priority)
Docs cover prototyping and mature systems but not the painful middle ground where AI is integrated but not yet valuable. How long does this phase last? When to abandon?
